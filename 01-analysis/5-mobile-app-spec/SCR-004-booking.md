## Обзор

Экран собирает параметры брони (число мест, вариант экипировки на каждое место, итоговую цену) и фиксирует запись на заезд. Это **самый сложный экран** клиентского сценария: на нём пересекаются два независимых лимита — места группы (`free_seats`, потолок конфигурации) и прокатный фонд экипировки (`free_rental_gear`) — и обрабатываются конкурентные записи (гонка / овербукинг).

Экран — **последний шаг перед подтверждением** в потоке записи: список заездов → карточка заезда → оформление → подтверждение. Соблюдается правило **≤ 3 экранов до подтверждения** (NFR-2). Таб-бар скрыт — это вложенный экран потока.

Оплата офлайн: экран показывает итоговую цену и фиксирует запись, онлайн-оплаты нет (как указано в брифе заказчика).

### User Story

> Как **Клиент**, я хочу записаться на выбранный заезд, указать число мест (себя и до 4 гостей), выбрать для каждого места свою или прокатную экипировку (шлем + подшлемник), увидеть итоговую цену и подтвердить бронь, чтобы гарантированно занять место на заезде.

### Бизнес-ценность

- Замена записи через Telegram и экипировку маркером — самостоятельная запись клиента.
- Исключение двойных броней и овербукинга (главная боль заказчика, NFR-8) за счёт серверного арбитража лимитов при атомарном создании брони (гарантия «0 двойных броней» на стороне бэкенда).
- Прозрачность стоимости до подтверждения (цена места + отдельный тариф проката экипировки), офлайн-оплата.

## Входные данные

| Название | Тип | Возможные значения | Описание |
|----------|-----|-------------------|----------|
| `slotId` | Параметр навигации | UUID | Идентификатор заезда, переданный с SCR-003. |
| `slot` | Кэш / Состояние | объект `Slot` | Данные заезда, переданные/закэшированные с предыдущего экрана: `start_at`, `configuration` (с потолком `capacity_cap`), `marshal`, `free_seats`, `free_rental_gear`, `price`, `rental_price`, `status`. Источник всех чисел экрана — **без хардкода**. |
| `seats_count` | Состояние формы | `1…max_seats` | Число мест. По умолчанию `1`. Верхняя граница — см. LOGIC-002. |
| `equipment_choice[i]` | Состояние формы | `own` / `rental` | Для каждого места `i` (`1…seats_count`) — вариант экипировки. |
| `rental_count` | Состояние (производное) | `0…seats_count` | Число мест с `equipment_choice = rental`. Считается из переключателей. |
| `idempotencyKey` | Состояние (генерируется на клиенте) | UUID | Ключ идемпотентности, **обязателен** при `createBooking` (R-022). Генерируется один раз на попытку отправки формы и переиспользуется при повторе после сетевого сбоя (E4, NFR-9). |

> Числовые лимиты и цены (`free_seats`, `free_rental_gear`, `route.capacity_cap`, `price`, `rental_price`) приходят из данных заезда и **не зашиваются** в макет. Единственная константа в группе — лимит **5 мест на бронь** (R-013).

**Инициализация формы.** При открытии экрана форма строится из переданного/закэшированного `slot`:
- `seats_count = 1` (по умолчанию — сам клиент).
- `equipment_choice[1]` — вариант по умолчанию по LOGIC-002: «Прокатная», если `free_rental_gear ≥ 1`, иначе «Своя».
- Данные заезда (дата/время, конфигурация, маршал, лимиты, цены) — из объекта `slot`.
- Полей профиля на экране **нет** — форма профилем (имя, телефон) **не предзаполняется**.

| Логика | Элемент/Триггер | Описание |
|--------|-----------------|----------|
| LOGIC-002 Расчёт доступности | Степпер «Число мест», переключатели «Своя/Прокатная экипировка», доступность CTA | Два независимых лимита: `1 ≤ seats_count ≤ min(free_seats, route.capacity_cap, 5)` и `rental_count ≤ free_rental_gear`. Пересчёт в реальном времени; строка «Прокат экипировки» скрыта при `rental_count = 0`. Итог созданной брони — серверный `price_total` ответа `createBooking` (R-005). |

## Инициализация

Экран открывается с уже полученными данными заезда (`slot`), переданными с SCR-003. При открытии **сетевые запросы не отправляются** — форма строится из переданного `slot`. Создание брони (`createBooking`) — это запрос **по действию** (тап CTA), а не при инициализации.

Опционально, при возврате к экрану или для актуализации лимитов перед показом, может выполняться повторный `getSlot(slotId)` — но это не блокирующая инициализация; базовый сценарий использует переданные данные (P4 — воспринимаемая скорость, без лишних ожиданий).

## Используемые запросы

> REST. GraphQL не используется.

### getSlot

**Тип:** REST  
**Метод:** GET  
**Спецификация:** `../api/slots/api.yaml` → `getSlot`

**Триггер:** Инициализация (только если `slot` не передан или нужна актуализация лимитов)

**Параметры:**

| Параметр | Тип | Обязательность | Источник | Описание |
|----------|-----|----------------|----------|----------|
| `slotId` | string (uuid) | Да | Параметр навигации | Идентификатор заезда. |

**Обработка ответа:**

| Результат | Условие | UI-реакция |
|-----------|---------|------------|
| Загрузка | — | Скелетон-шиммер формы |
| Успех 200 | `status = scheduled`, `free_seats > 0` | Построить форму; пересчитать лимиты (LOGIC-002) |
| Успех 200 | `free_seats = 0` или `status = cancelled` | Состояние «нет мест / заезд недоступен», CTA скрыт, кнопка «Вернуться к заездам» |
| 4xx с `message` | тело `Error` с непустым `message` | Снек с текстом из `message` (00-foundations §6 / LOGIC-008 Шаг 6) |
| 5xx | `internal_error` | Error-state первичной загрузки: заглушка + кнопка «Обновить» (повтор `getSlot`). Это **не** снек действия. |
| Сеть | Нет соединения / таймаут | Error-state первичной загрузки: заглушка «Не удалось загрузить. Проверьте соединение и попробуйте снова.» + кнопка «Обновить» |

### createBooking

**Тип:** REST  
**Метод:** POST  
**Спецификация:** `../api/bookings/api.yaml` → `createBooking`

**Триггер:** Тап на кнопку «Записаться» (CTA)

**Заголовки:**

| Заголовок | Тип | Обязательность | Источник | Описание |
|-----------|-----|----------------|----------|----------|
| `Idempotency-Key` | string (uuid) | **Да (обязателен, R-022)** | `idempotencyKey` из состояния | Обязательный ключ идемпотентности (NFR-9, R-022). Один и тот же ключ переиспользуется при повторе после сетевого сбоя (E4): повтор с тем же ключом и **тем же телом** → идентичный ответ ранее созданной брони (тот же 201), дубль не создаётся; тот же ключ с **другим телом** → 409 `idempotency_key_conflict`. Ключ хранится на сервере ≥ 24 ч. |

**Тело запроса** (`CreateBookingRequest`):

| Параметр | Тип | Обязательность | Источник | Описание |
|----------|-----|----------------|----------|----------|
| `slot_id` | string (uuid) | Да | `slotId` | Идентификатор заезда. |
| `seats_count` | integer `1…4` | Да | Состояние формы | Число мест. |
| `rental_count` | integer `0…seats_count` | Да | Производное из переключателей | Число прокатных комплектов экипировки. |

**Обработка ответа:**

| Результат | Условие (`code` в теле `Error`) | UI-реакция |
|-----------|--------------------------------|------------|
| Загрузка | — | CTA → индикатор загрузки; форма блокируется; повторный тап игнорируется (NFR-8) |
| Успех **201** | тело `CreateBookingResponse` (`Booking` + `price_total` + `is_first_booking` + `reminder_hours`) | Переход на BS-002 с объектом `booking` (итог берётся из серверного `price_total`, R-005) |
| **409 Conflict** | `slot_full` | **E1/E3** — нехватка мест/картов / гонка. Запись не создана. Прочитать `details.available_seats` → обновить `free_seats`, пересчитать лимиты, при необходимости откатить `seats_count`. Снек/нотис с текстом. CTA снова активен (если бронь снова валидна). |
| **409 Conflict** | `double_booking` | **E3** — двойная бронь (та же бронь уже создана). Запись не дублируется. Показать сообщение и увести на BS-002/«Мои записи» либо на SCR-003. |
| **409 Conflict** | нехватка прокатной экипировки (`details.available_rental_equipment < rental_count`) | **E2** — обновить `free_rental_gear`, предложить уменьшить число прокатных мест или переключить место(а) на «Свою». |
| **409 Conflict** | `idempotency_key_conflict` | Тот же `Idempotency-Key` повторно отправлен с **другим телом** (R-022). Защитный сценарий: сгенерировать новый `idempotencyKey` для изменённой формы; снек «Не удалось оформить запись. Попробуйте ещё раз.» CTA снова активен. |
| **410 Gone** | `slot_cancelled` | Заезд отменён организатором/центром (в т.ч. по погоде). Показать «Заезд отменён центром и больше недоступен.»; CTA скрыт; кнопка «Вернуться к заездам» → SCR-002. |
| **422 Unprocessable** | `slot_started` | Заезд уже стартовал. Показать «Заезд уже стартовал — запись недоступна.»; CTA скрыт; «Вернуться к заездам». |
| **400 Bad Request** | `bad_request` | Параметры вне диапазона (защитный сценарий — UI не должен такое отправлять). Снек «Не удалось оформить запись. Проверьте параметры.» |
| **401** | `unauthorized` | Снек «Сессия истекла, войдите снова», затем перевод на повторную авторизацию (SCR-001). |
| **Прочие 4xx с `message`** | любой `code`, тело `Error` с непустым `message` | Снек с текстом из `message` (общее правило 00-foundations §6 / LOGIC-008 Шаг 6). CTA возвращается в активное состояние. |
| **4xx без `message`** (дефолт) | тело `Error` без `message` | Снек «Не удалось выполнить. Попробуйте ещё раз.» (00-foundations §6). CTA возвращается в активное состояние. |
| **5xx / default** | `internal_error` | Снек «Что-то пошло не так. Попробуйте ещё раз позже.» (00-foundations §6). CTA возвращается в активное состояние. |
| **Сеть** | Нет соединения / таймаут | **E4** — «Не удалось выполнить. Проверьте соединение и повторите.» + кнопка «Повторить». Повтор отправляется с **тем же** `Idempotency-Key` (дубль не создаётся, NFR-9). До получения ответа CTA остаётся в загрузке. |

#### Матрица ошибок создания брони (код → HTTP → UX, R-023)

| `code` | HTTP | UX-реакция |

|---------|----------|-----------------|-----------|----------|
| Кнопка «−» | Уменьшить число мест | состояние `seats_count` | — | `seats_count -= 1`, убрать последнюю строку экипировки |
| Число | Текущее `seats_count` | состояние | — | — |
| Кнопка «+» | Увеличить число мест | состояние `seats_count` | — | `seats_count += 1`, добавить строку экипировки со значением по умолчанию |
| Подпись лимита | «Можно записать до K мест» (`K = max_seats`) | LOGIC-002 | — | — |

**Логика:**
- Степпер: LOGIC-002 — `max_seats = min(free_seats, route.capacity_cap, 5)`.
- При уменьшении `seats_count` лишние строки экипировки (с конца) убираются; при увеличении — добавляется строка со значением по умолчанию (см. блок 3).

**Условия доступности:**
- «−» disabled при `seats_count = 1` (минимум — сам клиент).
- «+» disabled при `seats_count = max_seats` (граница видна, без скрытой блокировки).

### 3. Блок выбора экипировки (на каждое место)

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|-----------------|-----------|----------|
| Строка «Место i» | «Место 1 (вы)», «Место 2 (гость)»… | состояние формы | — | — |
| Переключатель «Своя/Прокатная» | Сегментный, 2 опции | `equipment_choice[i]` | — | Переключить значение → пересчитать `rental_count` и цену |
| Счётчик проката | «Прокатной экипировки выбрано: `rental_count` из `free_rental_gear`» | LOGIC-002 | — | — |

**Логика:**
- Переключатели: LOGIC-002 — `rental_count ≤ free_rental_gear`.
- **Значение по умолчанию нового места:** «Прокатная», если `rental_count < free_rental_gear`; иначе «Своя».
- «Своя» занимает место в группе, но **не** уменьшает прокатный фонд; «Прокатная» уменьшает фонд (FR-14, UC-1 A1/A2).

**Условия доступности:**
- Опция «Прокатная» на ещё не-прокатных местах недоступна, когда `rental_count = free_rental_gear` (поясняющая подпись, E2). При `free_rental_gear = 0` «Прокатная» недоступна на всех местах.

### 4. Блок цены

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|-----------------|-----------|----------|
| Строка «Места» | `slot.price × seats_count` | `slot.price`, `seats_count` | — | — |
| Строка «Прокат экипировки» | `slot.rental_price × rental_count` (скрыта при `rental_count = 0`) | `slot.rental_price`, `rental_count` | — | — |
| «Итого» (превью) | `slot.price × seats_count + slot.rental_price × rental_count` | LOGIC-003 (R-010; итог брони — серверный `price_total`, R-005) | — | — |
| Напоминание об оплате | «Оплата на месте: наличные или перевод на карту.» | foundations §6 | — | — |

**Логика:**
- Цена: LOGIC-003. Пересчёт в реальном времени при изменении `seats_count` **и** варианта экипировки. Итог дублируется на CTA.

**Источник истины по цене (канон):**
- **На SCR-004 брони ещё нет** — итог в форме это **превью**, клиентский расчёт из актуальных тарифов заезда: `slot.price × seats_count + slot.rental_price × rental_count` (R-010; «своя» экипировка бесплатна).
- Актуальные `price` и `rental_price` берутся из текущего объекта `slot` (переданного с SCR-003 либо обновлённого после `getSlot`).
- **После создания брони источник истины — сервер:** ответ `createBooking` (201) содержит серверное поле `booking.price_total` (R-005), которое и показывается на BS-002 и далее.
- **Изменение цены между шагами.** Если при актуализации `getSlot` между открытием экрана и `createBooking` изменились `price`/`rental_price`, UI пересчитывает **превью** из свежего `slot` и обновляет сумму в блоке цены и на CTA; окончательный итог брони фиксирует сервер в `price_total` ответа `createBooking`.

### 5. CTA «Записаться»

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|-----------------|-----------|----------|
| Кнопка «Записаться · `Итого` ₽» | Фикс. нижняя, во всю ширину | итог из LOGIC-003 | — | Отправить `createBooking` |

**Момент валидации:** UI-ограничения применяются непрерывно (степпер/переключатели). Финальный арбитраж лимитов — на сервере при создании брони.

**Логика:**
- При тапе → перевести CTA в загрузку, заблокировать форму, сгенерировать (если ещё нет) `idempotencyKey`, отправить `createBooking` с заголовком `Idempotency-Key`.

**Условия доступности:**
- CTA enabled, когда бронь валидна по LOGIC-002: `1 ≤ seats_count ≤ max_seats` и `rental_count ≤ free_rental_gear`.
- CTA заблокирован от повторного тапа во время отправки (NFR-8).

### Диаграмма переходов

```mermaid
stateDiagram-v2
    [*] --> Content : onEnter (slot передан)

    Content --> Submitting : тап «Записаться» (форма валидна)
    Submitting --> Submitting : повторный тап игнорируется (NFR-8)

    Submitting --> Success : 201 Created
    Success --> [*] : переход на BS-002

    Submitting --> E1 : 409 slot_full + available_seats >= 1
    Submitting --> E2 : 409 + available_rental_equipment < rental_count
    Submitting --> E3 : 409 slot_full (мест нет) / double_booking
    Submitting --> SlotGone : 410 slot_cancelled
    Submitting --> SlotStarted : 422 slot_started
    Submitting --> E4 : нет сети / таймаут
    Submitting --> Content : 5xx / прочая ошибка (снек)

    E1 --> Content : откат seats_count, форма разблокирована
    E2 --> Content : уменьшить прокат / «Своя»
    E3 --> Content : лимиты обновлены (мест >= 1)
    E3 --> [*] : мест нет → «Вернуться к заездам» (SCR-002)
    E4 --> Submitting : «Повторить» (тот же Idempotency-Key)
    SlotGone --> [*] : «Вернуться к заездам» (SCR-002)
    SlotStarted --> [*] : «Вернуться к заездам» (SCR-002)

    Content --> [*] : «назад» → SCR-003 (без сохранения формы)
```

**Защита от двойной брони и овербукинга (NFR-8):**
- Повторный тап «Записаться» во время отправки заблокирован — второй запрос не уходит.
- Финальную проверку лимитов делает бэкенд при атомарном создании записи; UI-ограничения (§степпер/переключатели) — предупреждающие, не заменяют серверную проверку. При проигрыше гонки — E3.

**Идемпотентность повтора (NFR-9):**
- `Idempotency-Key` (UUID) генерируется один раз на попытку отправки и переиспользуется при повторе после сетевого сбоя — сервер по тому же ключу не создаёт дублирующую бронь.
