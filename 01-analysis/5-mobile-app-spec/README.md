## Соглашения

- **Платформа:** **нативное мобильное приложение** (iOS + Android) — клиент устанавливается из
  сторов. Это даёт доступ к нативным возможностям, на которые опираются ТЗ: системный push
  и хранение сессионного токена в защищённом хранилище ОС (Keychain / Keystore).
- **API:** REST, спецификации — в `../api/` (домены `auth`, `profile`, `slots`, `bookings`, `instructors`). В ТЗ указываются точные `operationId`, метод и путь.
- **Числа не хардкодятся:** потолки конфигураций трассы, прокатный фонд экипировки, цены, лимиты, длительность — всегда приходят из данных заезда/конфигурации/ответов API.
- **Дизайн-макеты Figma** (когда будут готовы) фиксируются в файле Figma «Апекс картинг приложение». В шапке каждого SCR-/BS-файла — ссылка на node-id. Сопоставление макетов с ТЗ и согласованные решения по расхождениям — в отчёте-ревью `../3-design-brief/design-review.md`.
- **BS-002 — экран успеха, не Bottom Sheet** (по аналогии с RR-D03): в макете это полноэкранное «Вы записаны» с двумя кнопками. ID `BS-002` сохранён для обратной совместимости ссылок; тип — «Экран».
- **Правило отмены клиентом:** 1 час до старта (выбрано как разумный баланс между защитой от простоя и удобством клиента; при необходимости корректируется на этапе детализации use cases).
- **Оценка маршала:** Phase 2 (R-001). В MVP — только заглушка/отсутствие BS-005; бриф учтён в roadmap.

## Критерии качества ТЗ

- Каждое ТЗ экрана содержит: User Story, Бизнес-ценность, полную навигацию, входные данные, применяемые логики, инициализацию (диаграмма + таблица запросов), обработку всех состояний (Loading/Content/Empty/Error), детальное описание элементов с валидацией и условиями доступности, критерии приёмки (позитивные + негативные + edge cases).
- Обязательно покрыты race conditions при бронировании, late/early cancel, cancel by center (R-008), отсутствие сети, нехватка мест/экипировки.
- Все формулировки — на русском, в стиле и терминологии брифа заказчика (Денис, картинг-центр «Апекс», заезд, маршал, экипировка, карта, конфигурация трассы).
