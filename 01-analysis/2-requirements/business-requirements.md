# Бизнес-требования

> Этап 2. Зачем бизнесу продукт: цели, метрики успеха, ограничения верхнего уровня.

> **Скоуп — клиентское мобильное приложение и API для него.** Бизнес-цели, обслуживаемые
> существующей инфраструктурой (экран владельца с расписанием, интерфейс маршала у трассы, оценки), 
> в этот список не входят.

> **О нумерации требований.** ID требований (BR-NN, M-NN) сохраняются стабильными для
> трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа.

> **Роли в текущей поставке.** В текущую поставку входит только роль **Клиент** (самостоятельная
> запись на заезды с мобильного приложения). Интерфейсы маршала и владельца
> — **вне скоупа текущей поставки**: они обеспечиваются существующей инфраструктурой. Это
> осознанное решение в рамках MVP-подхода и жёсткого срока.

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :-- | :-- | :-- | :-- |
| M-1 | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей). | ≥ 70% к концу первого сезона | BR-2 |
| M-2 | Число двойных броней / конфликтов по посадочным местам в картах. | 0 (полное устранение) | BR-1 |
| M-4 | Простой картов из-за поздних отмен (доля поздних отмен от записанных). | Снижение относительно текущего уровня (целевой ориентир ≤ 10%) | BR-5 |
| M-6 | Готовность к старту сезона. | Запуск в срок ~2 месяцев с критичным набором функций (просмотр слотов, запись, отмены) | BR-7 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — старт высокого сезона (~2 месяца); бюджет ограничен → строго MVP-подход. (Бриф)
- **Оплата офлайн на старте.** Онлайн-оплата вне границ скоупа. (Бриф)
- **Mobile-first.** Клиентское приложение — с телефона; данные получает из существующего бэкенда через API.
- **Вне скоупа.** Интерфейсы маршала/владельца, расписание-CRUD и админка обеспечиваются существующей инфраструктурой. Онлайн-оплата, программа лояльности для постоянных клиентов — направления развития.
- **Оценки маршалов — Phase 2.** Оценки инструкторов-маршалов осознанно вынесены в Phase 2 и не входят в MVP ради попадания в жёсткий срок (~2 месяца). Эта функция планируется к реализации после запуска критичного набора MVP.
