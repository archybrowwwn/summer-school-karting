# Функциональные требования

> Этап 2. Что система должна делать (функции, поведение).

Приоритеты по MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии времени; **Could** — приятное дополнение; **Won't** — вне границ скоупа.

## Границы скоупа (клиентское приложение + API)

Скоуп — **клиентское мобильное приложение и API для него**. Функции маршала, владельца (Дениса) и администратора обеспечиваются **существующей инфраструктурой**. 
- **Слоты (заезды) приходят из бэкенда через API.** Экранов управления расписанием в клиентском приложении нет.
- **Трассы и маршалы** доступны клиенту как справочные (read-only) данные.

## Просмотр и фильтрация заездов (клиент)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-9 | Система должна показывать клиенту список предстоящих слотов на **ближайшие 7 дней** (дефолт API) с составом: время старта, конфигурация трассы, маршал, всего мест (картов), свободно картов, цена. Заполненные и отменённые слоты не скрываются, а помечаются «Мест нет». | Must | R-027; Бриф |
| FR-38 | Система должна позволять фильтровать заезды по: дате, конфигурации трассы (короткая новичковая / длинная опытная), маршалу и наличию свободных картов. | Must | R-012 |
| FR-9a | Система должна показывать клиенту карточку заезда: дата/время, конфигурация трассы, описание брифинга, маршал, цена, всего/свободно картов, доступность прокатной экипировки, адрес картинг-центра. | Must | Бриф; RR-D07 |
| FR-44 | Система должна показывать в карточке заезда геолокацию картинг-центра: текстовый адрес и координаты на карте (Яндекс.Карты). При недоступности сети/карт показывается fallback (только текст). | Must | R-011 |

## Запись на заезд (клиент)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-10 | Система должна позволять клиенту записаться на выбранный слот заезда. | Must | Бриф |
| FR-11 | Система должна позволять клиенту при записи выбрать для каждого участника вариант экипировки (шлем, подшлемник): своя или прокатная. | Must | Бриф |
| FR-12 | Система должна позволять клиенту забронировать несколько картов (себя + гости) — **не более 5 картов на одну бронь** (R-013). Собирается только количество мест и выбор экипировки. | Must | R-013 |
| FR-13 | Система должна ограничивать бронь величиной `max_seats = min(free_seats, route.capacity_cap, 5)`, где `route.capacity_cap` — лимит трассы (новичковая ≤ 8 человек, опытная ≤ 14). Картов всего 14. | Must | Бриф (ограничения новичков); R-013 |
| FR-14 | Система должна вести **отдельный** учёт прокатной экипировки: `rental_count <= free_rental_gear`. Запись «со своей экипировкой» занимает карт, но не затрагивает фонд проката. | Must | R-015 |
| FR-15 | Система должна запрещать запись, превышающую лимиты (FR-13, FR-14), строго исключая запись двоих клиентов на один карт (овербукинг). | Must | Бриф (боль заказчика) |
| FR-45 | Система должна использовать `price_total` из ответа API как итоговую стоимость (с учётом прокатной экипировки, если она платная). | Must | R-005 |

## Мои бронирования и отмены (клиент)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-35a | Система должна показывать клиенту историю его бронирований (предстоящие, прошедшие, отменённые). | Must | R-025 |
| FR-16 | Система должна позволять отменить запись. Отмена возможна только **целиком** (освобождаются все занятые карты брони). | Must | R-014 |
| FR-17 | При ранней отмене (не позднее чем за 1 час до старта) система освобождает забронированные карты обратно в слот. Граница ранней отмены — ровно за 1 час включительно. | Must | Бриф; R-014 |
| FR-18 | При поздней отмене (< 1 часа до старта, включая "за 10 минут") система фиксирует статус «поздняя отмена» без освобождения места, чтобы избежать простоя (ограничить, но не злить). | Must | Бриф (отмена за 10 мин) |
| FR-46 | Система должна отображать заезды, отменённые центром (из-за сильного дождя или гололёда) со статусом «Отменён центром» и присылать push-уведомление. Повторная запись на отменённый заезд запрещена. | Must | Бриф (погода) |

## Профиль, Оплата и Уведомления

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-47 | Система должна позволять клиенту редактировать профиль и удалять аккаунт. | Must | R-009 |
| FR-30 | Оплата производится офлайн (наличные/перевод). Приложение лишь фиксирует запись. | Must | Бриф |
