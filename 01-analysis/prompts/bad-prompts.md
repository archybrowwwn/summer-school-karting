# «Плохие» промпты (антипримеры)

> Для лекции. Примеры, где ИИ выдаёт мусор: шаблонные user stories не из домена,
> выдуманные интеграции, путаница ролей и скоупа. Показываем для контраста с [good-prompts.md](good-prompts.md).

## Антипример 1: «Напиши ТЗ на приложение»

**Промпт:**
> Напиши ТЗ на мобильное приложение для картинга.

**Что не так:**
- Нет роли (клиент / админ / маршал) — ИИ добавит CRUD расписания и панель владельца.
- Нет формата выхода — получится простыня без ID требований.
- Нет границ MVP — ИИ предложит онлайн-оплату, рейтинги, CRM.

**Ответ ИИ (типичный фрагмент):**
> «Администратор создаёт слоты в календаре… Интеграция со Stripe… Рейтинг инструкторов 1–5 звёзд…»

**Как доводить:**
- Указать роль «только Клиент», формат таблицы, Won't-лист.
- Приложить бриф или ключевые факты (14 картов, офлайн-оплата, Telegram сейчас).

## Антипример 2: User stories «из головы»

**Промпт:**
> Сделай 15 user stories для e-commerce приложения спортивного центра.

**Что не так:**
- Домен «спортивный центр» слишком общий — stories про абонементы, тренеров, раздевалки.
- Нет терминов заказчика (заезд, маршал, экипировка, карта) — ломается трассировка к брифу.
- Нет числовых правил — ИИ придумает «до 10 человек» вместо cap трассы и фонда проката.

**Ответ ИИ (типичный фрагмент):**
> «Как клиент, я хочу купить годовой абонемент… Как администратор, я хочу редактировать расписание залов…»

**Как доводить:**
- Дать бриф «Апекс», зафиксировать `max_seats = min(free_seats, route.capacity_cap, 5)`.
- Просить только роль Клиент и ссылки на FR-*.

## Антипример 3: Модель данных без границ API

**Промпт:**
> Спроектируй БД для картинг-центра: все таблицы, индексы, триггеры, админские роли.

**Что не так:**
- Смешаны клиентский API и внутренняя админка — scope creep.
- ИИ проектирует полную схему БД вместо ресурсной модели API (R-004: слоты read-only).
- Появляются сущности вне MVP: Payment, Review, WeatherJob, AdminUser.

**Ответ ИИ (типичный фрагмент):**
> `CREATE TABLE payments … CREATE TABLE admin_schedule_edits … TRIGGER on slot_insert …`

**Как доводить:**
- Явно: «ресурсная модель API, Client + Booking создаёт клиент, Slot read-only».
- Просить ERD mermaid + sequence для createBooking/cancelBooking, не DDL админки.
