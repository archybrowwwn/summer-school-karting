## 1. Назначение и контекст

Экран собирает параметры брони (число мест/пилотов, вариант экипировки на каждое место, итоговую цену) и
фиксирует запись. Это **самый сложный экран** клиентского сценария: на нём пересекаются два
независимых лимита (свободные карты в заезде и прокатный фонд экипировки) и обрабатываются конкурентные записи.

Экран — **последний шаг перед подтверждением** в потоке записи: список → карточка заезда →
оформление → подтверждение. Соблюдается правило **≤ 3 экранов до подтверждения** (NFR-2): от
[SCR-002](SCR-002-slot-list.md) до [BS-002](BS-002-booking-success.md) клиент проходит карточку заезда
([SCR-003](SCR-003-slot-card.md)) и этот экран, без лишних шагов.

> **Без хардкода.** Все числа (`free_seats`, `free_gear_sets`, потолок трассы, `price`,
> `rental_price`) приходят из данных заезда/трассы и **не зашиваются** в макет или тексты
> (см. [foundations §6](00-foundations.md)). Правила двух лимитов и формула цены собраны в одном
> месте — **§6.5**; остальные разделы ссылаются на него, не повторяя условия.

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения, см. [foundations §1](00-foundations.md) и [brief-karting.md](brief-karting.md)).
- **Контекст:** клиент находится прямо у трассы, вокруг шумно, он может торопиться перед брифингом — нужны крупные
  элементы и минимум действий (P1, P2).
- **Цель пользователя:** быстро записать себя (и при желании 1–5 гостей), выбрать экипировку (своя или прокат) на
  каждого, увидеть итоговую цену и подтвердить бронь на заезд.
- **Цель системы:** собрать корректную бронь в рамках двух лимитов (карты и экипировка), не допустить овербукинга и
  двойной брони, зафиксировать запись.

## 4. Контент-инвентарь

### 4.1 Поля брони (вводимые / производные)

| Поле | Тип | Источник / правило | Примечание |
|------|-----|--------------------|------------|
| `seats_count` | Ввод | Число мест (пилотов): себя + до 4 гостей, итого **1–5** (без ввода имён гостей) | Минимум 1 (сам клиент), максимум — см. §6.5 |
| `gear_choice[i]` | Ввод | Для каждого места `i`: `own` / `rental` | Лейблы: «Своя» / «Прокатная» |
| `rental_count` | Производное | Количество мест с `gear_choice = rental` | Не вводится напрямую, считается из переключателей |
| `own_count` | Производное | `seats_count − rental_count` | — |
| Итоговая цена | Производное | `price × seats_count + rental_price × rental_count` | Пересчёт в реальном времени; формула — §6.5 |

> **Имена гостей не вводятся** — достаточно общего числа мест (NFR-2/NFR-3, минимум полей для быстрого ввода у трассы). Цена заезда
> (`price`) и **отдельный тариф проката экипировки** (`rental_price`) приходят из данных бэкенда; итог зависит
> и от числа прокатных мест (`rental_count`). Полная формула — §6.5.

### 4.2 Контекст заезда (read-only, для расчётов и подписей)

| Поле | Назначение в UI |
|------|------------------|
| Дата / время старта, конфигурация трассы, маршал | Краткая шапка-сводка заезда (что бронируем) |
| `free_seats` | Свободно картов в заезде — верхняя граница `seats_count` |
| `free_rental_gear` | Свободно комплектов экипировки — верхняя граница `rental_count` |
| Потолок трассы | Максимум мест для конфигурации (8 для новичковой, 14 для опытной); участвует в `min(...)` |
| `price` | Цена за заезд (один пилот) |
| `rental_price` | Тариф проката за один комплект экипировки (шлем, подшлемник) |

## 5. Структура и иерархия

Каркас — по [foundations §4.1](00-foundations.md): фикс. хедер → скролл-контент → фикс.
нижний CTA. Сверху вниз:

1. **Хедер** — «назад» + заголовок «Оформление записи».
2. **Сводка заезда** (компактная, read-only) — дата/время, трасса, маршал-инструктор; подпись о
   свободных картах и экипировке.
3. **Счётчик пилотов** (`seats_count`) — степпер 1–5 с подписью текущего лимита.
4. **Блок выбора экипировки по каждому месту** — список из `seats_count` строк: «Место 1 (вы)»,

5. **Блок цены** — цена за заезд × число пилотов + тариф проката × число комплектов = итог;
   напоминание об офлайн-оплате (формула — §6.5).
6. **Фикс. нижний CTA «Записаться»** — всегда виден, не перекрыт клавиатурой.

## 6. Компоненты и поведение

### 6.1 Степпер «Количество пилотов» (`seats_count`)

- Крупный степпер с кнопками **«−» / «+»** и числом по центру (тач-зоны ≥ 44–48 pt,
  [foundations §3.1](00-foundations.md)).
- **Минимум — 1** (клиент всегда бронирует карт себе): на минимуме «−» disabled.
- **Максимум** — по правилу лимита мест из **§6.5** (`min(free_seats, route.capacity_cap, 5)`).
  На максимуме «+» disabled (ограничение сверху, без скрытой блокировки) — клиент сразу видит доступную границу.
- Подпись под степпером поясняет текущий лимит словами (число — из актуальных данных, без хардкода).
- При **уменьшении** `seats_count` лишние строки выбора экипировки убираются; при
  увеличении — добавляются новые строки со значением по умолчанию (см. §6.2).

### 6.2 Переключатели «Своя / Прокатная» (на каждого пилота)

- На каждого пилота — сегментный переключатель из двух опций: **«Своя»** (экипировка клиента) / **«Прокатная»**
  (шлем и подшлемник центра). Состояние дублируется не только цветом — формой/текстом
  ([foundations §3.2](00-foundations.md)).
- **Значение по умолчанию для нового места:** «Прокатная», если `rental_count < free_rental_gear`;
  иначе «Своя» (чтобы не создавать заведомо невалидную бронь при нехватке шлемов).
- **Ограничение сверху по прокату:** суммарно нельзя выбрать «Прокатная» больше, чем
  `free_rental_gear`. При достижении границы опция «Прокатная» на еще не-прокатных строках
  становится **недоступной** с поясняющей подписью (см. §8, E2), а не молча игнорируется.
- Подпись-счётчик «Комплектов выбрано: `rental_count` из `free_rental_gear`» обновляется в реальном времени.

### 6.3 Блок итоговой цены

- Показывает две строки — **заезд** (`price × seats_count`) и, если выбрана экипировка центра,
  **прокат** (`rental_price × rental_count`) — и **итог** крупно и контрастно ([foundations §3.2](00-foundations.md)). Формула — §6.5.
- **Пересчёт в реальном времени** при изменении `seats_count` **и** варианта экипировки (аренда шлема влияет
  на итог). Строка «Прокат» полностью скрыта при `rental_count = 0`. Итог дублируется суммой на кнопке CTA.
- Под ценой — напоминание об офлайн-оплате (текст из [foundations §6](00-foundations.md):
  «Оплата на месте: наличные или перевод на карту.»).

### 6.4 Кнопка «Записаться» (нижний CTA)

- Во всю ширину контентной области, всегда видна над экранной клавиатурой ([foundations §4.1](00-foundations.md)).
- **Enabled**, когда бронь полностью валидна по правилам лимитов из **§6.5**.
- **Disabled** при невалидной комбинации (с понятной поясняющей микрокопией рядом).
- **Состояние отправки (loading):** при тапе CTA переходит в индикатор загрузки и
  **блокируется от повторного клика** (защита от двойных броней и овербукинга, NFR-8 — см. §7).

### 6.5 Правила лимитов и цены (единый источник, FR-13–FR-15, FR-30)

**Лимиты (два независимых):**

| Лимит | Граница | Правило |
|-------|---------|---------|
| Число пилотов `seats_count` | `1 ≤ seats_count ≤ min(free_seats, route.capacity_cap, 5)` | Минимум 1 (сам клиент); максимум — меньшее из свободных картов, потолка трассы (8 или 14) и 5 (R-013). |
| Прокатная экипировка `rental_count` | `rental_count ≤ free_rental_gear` | Считается из переключателей «Прокатная»; не может быть больше свободных шлемов на складе. |

- **«Своя экипировка»** занимает карт в заезде, но **НЕ** уменьшает прокатный фонд шлемов.
- **«Прокатная экипировка»** занимает карт в заезде **И** уменьшает прокатный фонд экипировки.
- Следствие: при `free_rental_gear = 0` запись на заезд возможна **только со своим шлемом**.

**Формула итоговой цены:**

```
Итого = price × seats_count + rental_price × rental_count
```

- `price` — цена за заезд пилота, `rental_price` — тариф за прокат одного комплекта (оба значения приходят из API).
- Прокат экипировки тарифицируется **отдельно**, поэтому итог зависит от числа прокатных мест.

Эти правила — **единственное место** их описания для экрана; разделы §6.1, §6.2, §6.3, §8 ссылаются сюда.

## 7. Состояния экрана

Базовый паттерн состояний — [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана). Ниже описана специфика экрана SCR-004.

| Состояние | Содержимое / Поведение |
|-----------|------------------------|
| **Loading (первичный)** | Показывается skeleton-заглушка в форме будущих блоков (каркас сводки заезда, степпер, строки экипировки). Кнопка CTA заблокирована. |
| **Content (активное)** | Отображаются все элементы управления (§5). Бронь валидна по правилам §6.5, кнопка CTA активна и содержит итоговую сумму. |
| **Loading (отправка)** | При тапе на CTA кнопка переходит в состояние загрузки (спиннер). Все элементы ввода (степпер, переключатели) блокируются (`disabled`), чтобы исключить изменение параметров во время запроса. Защита от повторного клика (NFR-8). |
| **Error (сетевая)** | Если запрос бронирования завершился сетевой ошибкой, показывается системный диалог/шторка ошибки с кнопкой «Повторить» (использует сохранённый на клиенте `Idempotency-Key`). Данные формы не сбрасываются. |

## 8. Валидации и микрокопия

Экран обрабатывает три критические ошибки конкурентной записи и нехватки ресурсов прямо перед бронированием. Тон сообщений — уважительный, без обвинений (принцип P6, [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).

### E1: Превышение лимита экипировки в процессе выбора
- **Условие:** Клиент пытается переключить экипировку для очередного места в режим «Прокатная», но текущий `rental_count` уже равен `free_rental_gear` (весь свободный фонд шлемов на этот слот исчерпан).
- **Поведение UI:** Опция «Прокатная» для остальных строк становится `disabled`.
- **Микрокопия:** Под блоком выбора экипировки выводится текст: *«Все доступные комплекты экипировки центра на этот заезд уже заняты. Вы можете записать следующих пилотов только со своим шлемом.»*

### E2: Конкурентный овербукинг мест (Ошибка API при отправке)
- **Условие:** Пока клиент оформлял запись, другие пользователи заняли карты, и `free_seats` стал меньше выбранного `seats_count`. API возвращает ошибку `409 Conflict (ERR_SLOT_FULL)`.
- **Поведение UI:** Экран блокирует CTA, обновляет счётчик мест и показывает сообщение. Кнопка возврата к списку становится явной (см. §11.3).
- **Микрокопия:** *«Извините, пока вы оформляли бронь, свободные карты закончились. На этот заезд мест больше нет.»*

### E3: Конкурентный овербукинг экипировки (Ошибка API при отправке)
- **Условие:** Пока клиент оформлял запись, шлемы забронировали для другого заезда, `free_rental_gear` уменьшился. API возвращает ошибку `409 Conflict (ERR_GEAR_DEPLETED)`.
- **Поведение UI:** Счётчик комплектов экипировки обновляется, переключатели сбрасываются в «Своя», и выводится предупреждение.
- **Микрокопия:** *«Не удалось забронировать экипировку центра — комплекты были заняты другим пилотом. Доступна запись только со своим шлемом.»*

## 9. Доступность и mobile-first

- **Интерактивные зоны:** Кнопки степпера «−»/«+» и сегменты переключателей имеют тач-зону ≥ 44×44 pt ([foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры)).
- **Контрастность при ярком солнце:** Итоговая стоимость и кнопка CTA используют высококонтрастные цвета ([foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)), легко считываемые на улице у трека.
- **Доступность состояний:** Выбранное состояние экипировки («Своя»/«Прокатная») обозначается не только цветом фона сегмента, но и жирным начертанием текста и контрастной пиктограммой-точкой.
- **Экранная клавиатура:** Поля ввода текста отсутствуют (только степпер и переключатели), что полностью исключает перекрытие интерфейса клавиатурой на экранах малого формата (iPhone SE).

## 10. Критерии приёмки дизайна

```gherkin
AC-001: Выбор количества пилотов в пределах лимита
  Дано клиент находится на экране оформления записи (SCR-004)
  Когда клиент нажимает «+» в степпере количества пилотов
  Тогда значение увеличивается на 1
  И в блоке экипировки добавляется новая строка для гостя с дефолтным значением
  И итоговая стоимость пересчитывается в реальном времени

AC-002: Блокировка степпера на верхней границе картов
  Дано заезд имеет свободными 2 карта
  Когда клиент открывает экран оформления
  Тогда кнопка «+» в степпере заблокирована (`disabled`) при достижении значения 2
  И под степпером отображается текст «Можно записать до 5 пилотов»

AC-003: Ограничение выбора прокатной экипировки
  Дано на складе свободно 1 комплект экипировки
  И клиент выбрал количество пилотов: 2
  Когда клиент выбирает «Прокатная» для Места 1
  Тогда для Места 2 опция «Прокатная» блокируется (`disabled`)
  И под блоком появляется предупреждение о нехватке шлемов (E1)

AC-004: Расчёт итоговой стоимости
  Дано цена заезда 2000 ₽, цена проката экипировки 300 ₽
  Когда клиент выбирает 2 пилота и для обоих указывает «Прокатная»
  Тогда в блоке цены отображается: Заезд: 4000 ₽, Прокат: 600 ₽, Итого: 4600 ₽
  Когда клиент меняет у одного пилота выбор на «Своя»
  Тогда строка Прокат меняется на 300 ₽, а Итого становится 4300 ₽

AC-005: Защита от двойной отправки формы
  Дано клиент заполнил форму и нажал «Записаться»
  Тогда кнопка CTA переходит в состояние Loading
  И вся форма (степпер, переключатели) блокируется для ввода
  И отправляется один запрос с уникальным `Idempotency-Key`

AC-006: Обработка овербукинга карт на бэкенде
  Дано клиент отправил форму записи на 2 места
  Когда API возвращает ошибку `ERR_SLOT_FULL` (карты заняли)
  Тогда на экране отображается сообщение E2
  И кнопка CTA меняет текст на «Вернуться к списку»
```

---

## 11. Решения

1. **Имена гостей — не вводятся.** Для бронирования на уличной трассе картинг-центра достаточно указать общее число мест (степпер 1–5) без заполнения ФИО гостей. Это сокращает время оформления до нескольких тапов (принципы NFR-2/NFR-3). См. §4.1, §6.5.
2. **Цена проката — отдельный тариф.** Прокат шлема и экипировки (`rental_price`) тарифицируется отдельно от стоимости самого заезда. Сумма считается динамически: `price × seats_count + rental_price × rental_count`. См. §6.5.
3. **Поведение при E2 с нулём мест.** При получении ошибки овербукинга карт (`ERR_SLOT_FULL`) приложение показывает обновлённые данные и **явную кнопку** возврата к списку слотов. Автоматический переход назад запрещён, чтобы клиент успел прочитать причину отмены. См. §8 (E2).
4. **Идемпотентный ключ записи генерирует клиент.** UUID `Idempotency-Key` создаётся на стороне мобильного приложения один раз при первой попытке отправки формы и переиспользуется при повторах (в случае сетевых сбоев E4). Бэкенд не генерирует этот ключ, отдельного UI под него нет. Решение согласовано с sequence-диаграммой и US-14.
