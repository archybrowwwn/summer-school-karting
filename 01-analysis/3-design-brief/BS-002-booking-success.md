## 1. Назначение и контекст

Шторка завершает основной сценарий записи: **подтверждает, что бронь на заезд успешно создана**, даёт
клиенту наглядную сводку того, на какой заезд он записался, напоминает об **офлайн-оплате** и
предлагает дальнейший шаг — посмотреть запись в «Моих заездах» или вернуться к слотам.

BS-002 — это **финальный, успешный** результат потока записи. Она открывается **после
успешного подтверждения** на экране оформления [SCR-004](SCR-004-booking.md). Ошибки записи
(нехватка картов, превышение лимита участников в 8 человек для новичковой трассы, гонка запросов,
сетевой сбой) обрабатываются **на SCR-004** и до этой шторки не доходят — см. [UC-1](../2-requirements/use-cases.md).
Существующий бэкенд выступает как black-box источник истины (R-004), гарантируя отсутствие двойных
броней до открытия шторки. Таким образом, BS-002 закрывает путь записи в рамках принципа
«короткий путь к записи» (см. [foundations §2, P2](00-foundations.md#2-дизайн-принципы)).

## 2. Пользователь и цель

- **Пользователь:** клиент, только что оформивший запись на заезд (роль — единственная в рамках
  текущей поставки, см. R-028, [foundations §1](00-foundations.md#1-продукт-и-аудитория)). Часто находится
  прямо у трассы, вокруг шумно, спешит, хочет убедиться, что «всё получилось», и понять, что делать дальше.
- **Цель пользователя:** получить однозначное подтверждение успеха, проверить ключевые параметры
  брони (когда, какая конфигурация трассы, сколько картов забронировано, нужна ли экипировка) и понять способ оплаты.
- **Цель продукта:** закрыть петлю записи спокойным позитивным завершением (без давления), снизить
  тревогу «записался ли я», мягко подвести к разделу «Мои заезды» ([US-5](../2-requirements/user-stories.md)).

## 3. Навигация

### Входящие переходы
- **SCR-004 → BS-002** — после успешного подтверждения записи на экране
  [Оформление записи](SCR-004-booking.md) (бронь создана на бэкенде, доступные карты атомарно уменьшены).
  Это единственная точка входа.

### Исходящие переходы
- **«Мои заезды» → [SCR-005](SCR-005-my-bookings.md)** — переход к списку своих записей;
  созданная бронь отображается там со всеми деталями.
- **«Готово» / закрытие → возврат к списку слотов [SCR-002](SCR-002-slot-list.md)** — шторка
  закрывается, клиент оказывается на корневой вкладке «Заезды» (стек оформления записи
  завершён). Промежуточные экраны [SCR-003](SCR-003-slot-card.md)/[SCR-004](SCR-004-booking.md)
  в стеке не остаются — путь записи окончен.

### Свойства шторки
Единые правила bottom sheet (высота по контенту, бэкдроп, грабер, swipe-to-close, анимация,
расположение кнопок) — см. [foundations §4.3](00-foundations.md#43-bottom-sheet-шторки-bs-001--bs-002--bs-003).
Специфика BS-002 — в §6.

## 4. Контент-инвентарь

Все значения подставляются из данных созданной брони/слота; числа и лимиты **не хардкодятся**
(см. [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).

| # | Элемент | Источник данных | Назначение |
|---|---------|-----------------|------------|
| 1 | Иконка успеха | — | Невербальный сигнал «готово» (не только цветом, см. §9). |
| 2 | Заголовок «Вы записаны на заезд» | — | Однозначное подтверждение успеха. |
| 3 | Дата и время старта | `slot.start_at` | Когда начнется заезд (включая предстартовый инструктаж). |
| 4 | Конфигурация трассы | `slot.track.configuration` | Короткая трасса (для новичков) / Длинная трасса (для опытных). |
| 5 | Маршал-инструктор | `slot.marshal.name` | Кто ведёт заезд, проводит брифинг и следит за безопасностью. |
| 6 | Число забронированных картов | `booking.karts_count` | Сколько картов забронировано (всего до 14 на слот, до 8 на новичковые заезды). |
| 7 | Вариант экипировки | `own` / `rental` по картам | Наличие проката экипировки (шлем, подшлемник) или использование своей. |
| 8 | Итоговая цена | расчёт по слоту/брони | Сколько платить с учётом заезда и выбранного проката экипировки. |
| 9 | Напоминание об оплате | текст из foundations §6 | Способ оплаты — строго офлайн (наличные или перевод). |
| 10 | Кнопка «Мои заезды» | — | Переход к списку заездов [SCR-005](SCR-005-my-bookings.md). |
| 11 | Кнопка «Готово» / закрытие | — | Закрыть шторку → переход к списку слотов [SCR-002](SCR-002-slot-list.md). |

> **Вариант экипировки** отображается в сводке так, чтобы было понятно, включен ли прокат шлема
> и подшлемника (например, «Экипировка: свой шлем» или «Экипировка: прокат (2 комплекта)»). Точная компоновка
> остаётся за дизайнером; базовые лейблы берутся из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия).

## 5. Структура и иерархия

Сверху вниз:

1. **Грабер** (полоска для swipe-to-close).
2. **Блок успеха:** иконка успеха + заголовок-подтверждение.
3. **Сводка брони:** дата/время старта, конфигурация трассы, маршал, число картов, вариант экипировки,
   итоговая цена.
4. **Напоминание об оплате** (офлайн: наличные / перевод на карту).
5. **Кнопки действий** (внизу шторки): основная — **«Мои заезды»**, вторичная — **«Готово»** (закрытие).

### ASCII-wireframe

```
┌─────────────────────────────────────┐
│                ▭▭▭                    │  ← грабер (swipe-to-close)
│                                       │
│               ( ✓ )                   │  ← иконка успеха (форма + текст, не только цвет)
│        Вы записаны на заезд           │  ← заголовок-подтверждение
│                                       │
│  ┌─────────────────────────────────┐ │
│  │ 🗓  <start_at: дата · время>     │ │
│  │ 🏎  Трасса: <track.configuration> │ │  ← сводка брони заезда
│  │ 👤  Маршал: <marshal.name>       │ │
│  │ 👥  Картов: <karts_count>        │ │

│  │ ───────────────────────────────  │ │
│  │ 💳  Итого: <итоговая цена>        │ │  ← крупно, контрастно
│  └─────────────────────────────────┘ │
│                                       │
│  ⓘ Оплата на месте: наличные         │  ← напоминание об офлайн-оплате (Денис принимает так)
│     или перевод на карту.             │
│                                       │
│  ┌─────────────────────────────────┐ │
│  │          Мои заезды             │ │  ← основная кнопка → SCR-005
│  └─────────────────────────────────┘ │
│  ┌─────────────────────────────────┐ │
│  │           Готово                │ │  ← вторичная кнопка → закрыть → SCR-002
│  └─────────────────────────────────┘ │
└─────────────────────────────────────┘
░░░ бэкдроп (затемнение) ░░░
```

> Ключевые числа (старт, количество картов, цена) должны быть крупными и контрастными
> ([foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)).

## 6. Компоненты и поведение

### 6.1 Блок успеха
- Иконка успеха + короткий заголовок. Состояние «успех» передаётся **комбинацией формы/иконки и
  текста**, а не только цветом ([foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)).
- Тон — спокойный, дружелюбный, информирующий, без излишнего восклицательного давления
  ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).

### 6.2 Сводка брони
- Список «лейбл → значение» с параметрами заезда из §4. Один столбец (mobile-first layout).
- Итоговая цена выделена как самый визуально значимый элемент блока.
- Отображение экипировки показывает соотношение своей и прокатной. Если клиент указал, что все участники
  со своим шлемом и подшлемником, строка принимает вид «Своя экипировка». При `rental_count > 0` пишется
  количество прокатных комплектов.

### 6.3 Кнопки
- **«Мои заезды»** — основной CTA, растянутый во всю ширину контентной области, ведёт на
  [SCR-005](SCR-005-my-bookings.md).
- **«Готово»** — вторичная кнопка, закрывает шторку и возвращает на главный экран слотов
  [SCR-002](SCR-002-slot-list.md).
- Размеры и отступы тач-зон спроектированы крупными по
  [foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры). Это критично, так как приложением пользуются
  прямо у трассы, возможно, замёрзшими или влажными руками.

### 6.4 Запрос разрешения на push (после первой записи)
- **Только при первой успешной записи** клиента в приложении: сразу после показа сводки поверх контента
  или в стык предлагается включить напоминания о заезде. После согласия вызывается **системный запрос
  разрешения на push-уведомления** ([foundations §8.1](00-foundations.md#81-напоминания-и-уведомления-fr-33-nfr-13)).
- Это **единственная точка сбора push-согласий** в продукте. На экранах авторизации/регистрации
  ([SCR-001](SCR-001-registration.md)) запрос не выдаётся, чтобы не создавать барьеров.
- При последующих успешных записях запрос **повторно не отображается**. Отказ от push не блокирует бронь
  и не преследует пользователя навязчиво (P6 — без давления). Основная ценность для клиента: вовремя напомнить
  о заезде, чтобы карт не простаивал, а также мгновенно предупредить через push, если заезд будет
  **отменён центром по причине погоды** (R-008 — сильный дождь, гололёд на уличной трассе).

### 6.5 Поведение bottom sheet
- Высота рассчитывается по контенту, присутствует бэкдроп, грабер, жест swipe-to-close вниз и плавная
  анимация появления — по [foundations §4.3](00-foundations.md#43-bottom-sheet-шторки-bs-001--bs-002--bs-003).
- Тап по бэкдропу, свайп вниз или нажатие на «Готово» приводят к одному исходу — закрытию шторки с переходом
  на [SCR-002](SCR-002-slot-list.md). Поскольку бронь уже успешно зафиксирована на бэкенде, закрытие
  свободным жестом полностью безопасно.
- Если у пользователя забронировано много мест или выбран сложный состав экипировки, контент скроллится
  внутри шторки вертикально, при этом блок кнопок может оставаться зафиксированным внизу.

## 7. Состояния экрана

Шторка BS-002 инициализируется исключительно по факту успешно созданной и подтверждённой бэкендом записи.
Основное и единственное состояние — **Content (успех)**.

| Состояние | Применимость | Примечание |
|-----------|--------------|------------|
| **Content (успех)** | Да | Отображение сводки созданной брони на заезд + управляющие кнопки. Целевой режим. |
| **Loading** | Нет (в шторке) | Индикация ожидания ответа бэкенда происходит на экране [SCR-004](SCR-004-booking.md) до инициализации шторки. |
| **Empty** | Нет | Исключено: шторка не может быть вызвана без валидных данных созданной брони. |
| **Error** | Нет (в шторке) | Все ошибки (нет картов, превышен новичковый лимит в 8 человек, сетевой сбой) перехватываются и обрабатываются на [SCR-004](SCR-004-booking.md) — см. [UC-1](../2-requirements/use-cases.md). В BS-002 ошибок не бывает. |

Общий паттерн состояний — см.
[foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана).

## 8. Валидации и микрокопия

Экран является чисто информационным и не содержит полей ввода, поэтому валидации на клиенте отсутствуют.
Контентные правила:

| Правило | Описание |
|---------|----------|
| Напоминание об оплате | Строго регламентированный текст из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия): **«Оплата на месте: наличные или перевод на карту.»** Запрещено заменять синонимами. |
| Лейблы экипировки | «Свой шлем/подшлемник» / «Прокат экипировки» — в зависимости от выбора на SCR-004. |
| Отражение в списках | После успешного закрытия шторки бронь обязана мгновенно появиться в разделе «Мои заезды» [SCR-005](SCR-005-my-bookings.md) (R-028 / US-5). |
| Пересчёт доступности | Свободный фонд картов на бэкенде уменьшается на `karts_count`. При следующем обновлении экранов [SCR-002](SCR-002-slot-list.md)/[SCR-003](SCR-003-slot-card.md) данные о доступных картах обновляются. |
| Тон микрокопии | Уважительное обращение на «вы», отсутствие восклицательных знаков и давления (foundations §6). |
| Подстановка переменных | Дата, конфигурация трассы, имя маршала и итоговая цена берутся из объекта ответа API без хардкода. |

## 9. Доступность и mobile-first

Экран спроектирован с учётом жестких условий эксплуатации на картодроме (солнце, спешка, перчатки, мокрые руки):

- **Габариты кнопок:** «Мои заезды» и «Готово» имеют высоту не менее 48dp, тач-зоны соответствуют
  [§3.1](00-foundations.md#31-тач-зоны-и-размеры).
- **Визуальная доступность:** Успех подчёркивается формой галочки и текстом «Вы записаны на заезд»,
  что обеспечивает читаемость на ярком солнце у открытой трассы даже при плохой контрастности экрана ([§3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)).
- Ключевые данные (время старта, трасса, итоговая стоимость) выделены увеличенным кеглем.
- Все элементы размечены для экранных дикторов (Screen Reader); текстовый слой полностью дублирует
  смысл графических пиктограмм.
- Адаптивная вёрстка: при экстремальном системном увеличении размера шрифта (Accessibility font scaling)
  текст не обрезается, а переносится, шторка включает внутренний вертикальный скролл.

## 10. Критерии приёмки дизайна

```gherkin
Функция: Подтверждение успешной записи на заезд (BS-002)

  Сценарий: AC-001 — после успешной записи на картинг показана сводка
    Дано клиент нажал кнопку подтверждения на экране SCR-004
    И бэкенд подтвердил создание брони без двойных записей (R-004)
    Когда открывается шторка BS-002
    Тогда отображается блок успеха с иконкой и текстом "Вы записаны на заезд"
    И видна сводка: дата и время старта, конфигурация трассы (короткая/длинная), маршал, количество картов, тип экипировки и итоговая цена
    И все данные динамически подставлены из ответа API

  Сценарий: AC-002 — отображение напоминания об оплате
    Дано открыта шторка BS-002
    Тогда под сводкой отображается фиксированный текст: "Оплата на месте: наличные или перевод на карту."

  Сценарий: AC-003 — переход в раздел «Мои заезды»
    Дано открыта шторка BS-002
    Когда клиент нажимает на кнопку "Мои заезды"
    Тогда шторка закрывается и происходит переход на экран SCR-005

  Сценарий: AC-004 — обновление данных в системе после закрытия
    Дано бронь успешно зафиксирована в BS-002
    Когда клиент переходит в «Мои заезды» (SCR-005)
    Тогда созданная запись отображается в активных заездах
    И на экранах SCR-002/SCR-003 количество свободных картов на этот слот уменьшено на забронированное число

  Сценарий: AC-005 — закрытие шторки по кнопке «Готово» или жестом
    Дано открыта шторка BS-002
    Когда клиент нажимает "Готово" или делает свайп шторки вниз, или тапает по бэкдропу
    Тогда шторка плавно закрывается
    И пользователь оказывается на экране списка слотов SCR-002 (промежуточные экраны удалены из стека)

  Сценарий: AC-006 — отображение состава экипировки
    Дано в брони присутствует прокат экипировки (rental)
    Когда шторка BS-002 отображает контент
    Тогда в строке "Экипировка" выводится количество прокатных комплектов (шлем + подшлемник)
    И в итоговой цене учитывается стоимость проката по тарифу из API (R-015)

  Сценарий: AC-007 — запрос разрешений на push-уведомления при первой записи
    Дано это самая первая успешная запись клиента в приложении
    Когда открывается шторка BS-002
    Тогда пользователю демонстрируется предложение включить уведомления и вызывается системный диалог push
    И ценность объясняется напоминанием о заезде и алертами об отмене заезда центром по погоде (R-008)
    И при последующих бронированиях этот диалог больше не появляется
```

| # | Решение |
|---|---------|
| 1 | **Функции «Поделиться заездом» / «Календарь» в MVP отсутствуют.** Ограничиваем скоуп для ускорения запуска к началу высокого сезона (согласно брифу Дениса). |
| 2 | **Автоматическое закрытие по таймеру запрещено.** Экран закрывается только по явному действию пользователя, чтобы он гарантированно успел ознакомиться со способом офлайн-оплаты. |
| 3 | **Push-уведомления запрашиваются строго здесь.** Это идеальный контекст высокой лояльности после первой успешной записи. Позволяет возвращать клиентов и информировать о погодных отменах трассы (R-008). |
| 4 | **Правила отмены (проблема отмены за 10 минут) здесь не дублируются.** Данное ограничение и условия отмены выводятся на экране деталей заезда [SCR-006](SCR-006-booking-details.md) и в правилах бронирования. |
