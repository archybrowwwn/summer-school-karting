## 1. Назначение и контекст

Экран даёт клиенту **контроль над своими заездами** — отдельно **предстоящими** и
**прошедшими**. Это домашняя точка для проверки «куда и когда я записан», входа в детали
записи и (через детали) запуска отмены. Экран — корневой во вкладке «Мои записи» таб-бара
(см. [foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).

Контекст использования — уличный картинг-центр, шум трека, спешка перед выходом на трассу и брифингом
([brief-karting.md](brief-karting.md)): клиенту нужно **с одного взгляда**
понять ближайшую запись (когда старт, какая конфигурация трассы, сколько мест/картов забронировано) и её статус.

---

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения).
- **Цель пользователя:** быстро увидеть свои предстоящие заезды, убедиться в деталях
  ближайшей записи, при необходимости перейти к деталям/отмене; при отсутствии записей —
  понять, как записаться.
- **Покрываемая история:** [US-9](../2-requirements/user-stories.md) — «Как клиент, я хочу
  видеть список своих записей (предстоящих и прошедших), чтобы контролировать свои заезды».

## 3. Навигация

### Входящие
- **Тап по вкладке «Мои записи»** в таб-баре с любого корневого экрана АЗ
  ([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).
- Из шторки [BS-002 «Подтверждение записи»](BS-002-booking-success.md) по действию
  **«Мои бронирования»** (сразу после успешной записи на заезд).

### Исходящие
- **Тап по карточке заезда** → [SCR-006 «Детали брони + отмена»](SCR-006-booking-details.md).
- Переключение вкладок таб-бара → [SCR-002 «Список слотов»](SCR-002-slot-list.md),
  [SCR-007 «Профиль»](SCR-007-profile.md).
- Из **Empty-состояния** по действию «Записаться на заезд» → переход к списку слотов
  [SCR-002](SCR-002-slot-list.md).

### Каркас и таб-бар
Это **корневой** экран вкладки — таб-бар виден; на дочерних экранах (детали брони) скрывается
по правилу [foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона). Базовый каркас —
[foundations §4.1](00-foundations.md#41-базовый-каркас). Собственного нижнего CTA у экрана нет.

## 4. Контент-инвентарь

Данные — записи (брони) **текущего клиента**, см. [модель данных](../4-design/data-model.md).
Числовые лимиты не зашиваются в UI — берутся из данных слота/трассы
([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).

### 4.1 Карточка записи (booking)

| Поле | Источник | Назначение на карточке |
|------|----------|------------------------|
| `status` | booking | Бейдж статуса: `active` → «активен», `cancelled` → «отменен», `late_cancel` → «поздняя отмена». Текст + форма, не только цвет. |
| `start_at` | slot | Дата и время старта заезда — крупно, основной ориентир. |
| `route.name` | slot → track | Название конфигурации трассы (и её сложность/тип, если приходит). |
| `instructor.name` | slot → instructor | Имя маршала-инструктора. |
| `price` | slot / booking | Цена (итоговая по записи, включая прокат экипировки при наличии). Оплата офлайн — текст см. [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |
| `seats_count` | booking | Число мест (пилотов) в записи (себя + гости). |
| Вариант экипировки / `rental_count` | booking | Сводка по экипировке: своя / прокатная (сколько прокатных комплектов). Лейблы — [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |

> Карточка — **сводка**, достаточная для быстрого опознания заезда; полная детализация — на
> [SCR-006](SCR-006-booking-details.md).

### 4.2 Разделение списка
- **Предстоящие** — **только активные** записи (`status = active`), у которых `start_at` в будущем
  (актуальный ориентир и приоритетная секция).
- **Прошедшие** — записи с `start_at` в прошлом **И все отменённые центром или клиентом** (`cancelled` / `late_cancel`)
  независимо от даты старта. То есть отменённая по погоде или клиентом запись с будущим временем старта тоже попадает в «Прошедшие».

> Правило отнесения: запись активна и старт в будущем → «Предстоящие»; иначе (старт в прошлом
> **или** запись отменена) → «Прошедшие». Активность и время старта вычисляются из данных, без
> отдельного флага. Прошедшие сортируются **по убыванию даты старта** (свежие сверху).

## 5. Структура и иерархия

Сверху вниз: **хедер** (заголовок + переключатель/секции) → **скролл-список карточек заездов** →
**таб-бар**.

```
┌──────────────────────────────┐
│  Мои записи                  │  ← заголовок экрана (хедер)
│  [ Предстоящие | Прошедшие ] │  ← сегмент-переключатель (или секции)
├──────────────────────────────┤
│ ┌──────────────────────────┐ │  ← скролл-список карточек заездов
│ │ сб, 21 июн · 09:00       │ │     (карточка записи)
│ │ Длинная конфигурация     │ │
│ │ Маршал: Александр        │ │
│ │ 2 пилота · 1 прокатная   │ │
│ │ 4300 ₽          [активен]│ │  ← бейдж статуса (текст+форма)
│ └──────────────────────────┘ │
│ ┌──────────────────────────┐ │
│ │ вс, 22 июн · 18:00       │ │
│ │ Короткая трасса          │ │
│ │ Маршал: Дмитрий          │ │
│ │ 1 пилот · своя экипировка│ │
│ │ 2000 ₽          [активен]│ │
│ └──────────────────────────┘ │
├──────────────────────────────┤
│  Заезды · Мои записи · Профиль│  ← таб-бар
└──────────────────────────────┘
```

Группировка при варианте с секциями (вместо сегмента):

```
┌──────────────────────────────┐
│  Мои записи                  │
├──────────────────────────────┤
│  ПРЕДСТОЯЩИЕ                  │  ← заголовок секции (sticky)
│  [ карточка ]                │
│  [ карточка ]                │
│                              │
│  ПРОШЕДШИЕ                    │  ← заголовок секции (sticky)
│  [ карточка ]                │
│  [ карточка ]                │
└──────────────────────────────┘
```

Типографическая иерархия — [foundations §3.3](00-foundations.md#33-типографическая-иерархия-уровни-не-шрифты):
дата/время старта и итоговая цена — наиболее контрастные элементы карточки.

## 6. Компоненты и поведение

| Компонент | Поведение |
|-----------|-----------|
| **Переключатель «Предстоящие / Прошедшие»** | Сегмент-контрол вверху **или** разделение списка на sticky-секции (выбор паттерна — за дизайнером). При сегменте показывается одна группа за раз; активная вкладка визуально выделена (текст + состояние, не только цвет). По умолчанию открыты **Предстоящие**. |
| **Карточка записи (заезда)** | Кликабельна целиком (тач-зона ≥ [foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры)); тап → [SCR-006](SCR-006-booking-details.md). Крупная, в одну колонку, явно отделена отступами/границей. |
| **Бейдж статуса** | Текст + форма (контур/заливка/иконка), не только цвет — [foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1), [§3.5](00-foundations.md#35-иконки-и-индикаторы). Тексты — см. §8. |
| **Pull-to-refresh** | Жест «потянуть вниз» обновляет список заездов (актуальные статусы/данные из бэкенда) — паттерн [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана). |
| **Скролл-список** | Вертикальный, одна колонка; при варианте секций — sticky-заголовки групп. |

## 7. Состояния экрана

Применяется сквозной паттерн [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана).
Ниже — экранная специфика картинг-центра.

| Состояние | Содержимое / Поведение |
|-----------|------------------------|
| **Loading** | Скелетоны карточек записи (в форме будущего контента заездов), без блокирующего спиннера. |
| **Content** | Список карточек, разделённый на Предстоящие (только активные) / Прошедшие (прошлые заезды + все отменённые, в т.ч. по погоде). |
| **Empty (нет записей вообще)** | Заглушка «У вас пока нет записей» + действие **«Записаться на заезд»** → [SCR-002](SCR-002-slot-list.md). |
| **Empty в секции «Предстоящие»** (есть только прошедшие) | В секции/вкладке «Предстоящие» — короткая подсказка «Нет предстоящих записей» + «Записаться на заезд»; секция «Прошедшие» при этом доступна. |
| **Error** | Заглушка ошибки нейтральным тоном + кнопка **«Обновить»**. Текст сетевой ошибки — общий из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |

## 8. Валидации и микрокопия

### Доступ к данным
- Клиент видит **только свои записи** ([NFR-12](../2-requirements/non-functional-requirements.md),
  [foundations §8.2](00-foundations.md#82-безопасность-данных-в-ui-nfr-11-nfr-12), ограничение роли Клиент R-028). Чужих
  броней, контактов маршалов или административных данных на экране нет.

### Тексты бейджей статуса

| `status` | Текст бейджа |
|----------|--------------|
| `active` | «активен» |
| `cancelled` | «отменен» |
| `late_cancel` | «поздняя отмена» |

> Подробное пояснение причин отмены (отмена центром по погоде R-008 или поздний отказ клиента) — на
> [SCR-006](SCR-006-booking-details.md) / [BS-003](BS-003-cancel-confirm.md); на карточке —
> только короткий бейдж.

### Empty-состояние
- Заголовок: «У вас пока нет записей».
- Подсказка/действие: «Записаться на заезд» (ведёт на список слотов [SCR-002](SCR-002-slot-list.md)).
- Тон — спокойный, без давления ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).

### Лейблы экипировки и оплаты
- «Своя экипировка» / «Прокатная экипировка» (комплекты шлемов), текст об офлайн-оплате — переиспользуются из
  [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия), не вводятся заново.

## 9. Доступность и mobile-first

Базовые правила — [foundations §7](00-foundations.md#7-доступность-nfr-1) и
[§3](00-foundations.md#3-структурные-токены-без-бренда). Экранная специфика картинг-центра:

- **Крупные карточки** с большой тач-зоной — удобно тапать торопясь на треке, на ходу или в перчатках в зоне ожидания.
- **Читаемые статусы заездов**: бейдж дублирует смысл текстом и формой, легко считывается на ярком солнце у открытой трассы и
  при дальтонизме.
- **Читаемые даты/время**: формат старта понятен с первого взгляда (день недели + дата +
  время), без двусмысленности.
- Переключатель/секции и карточки заездов имеют доступные имена для screen reader; фокус-состояния
  элементов списка обязательны.

## 10. Критерии приёмки дизайна

```gherkin
Сценарий: AC-001 Разделение на предстоящие и прошедшие заезды
  Дано у клиента есть записи на заезды с разными датами старта и статусами
  Когда он открывает экран «Мои бронирования»
  Тогда записи разделены на группы «Предстоящие» и «Прошедшие»
  И по умолчанию отображаются предстоящие записи заездов
  И в «Предстоящие» попадают только активные записи с будущим временем старта

Сценарий: AC-001a Отменённый заезд с будущим стартом — в «Прошедшие»
  Дано запись отменена центром по погоде или клиентом (cancelled или late_cancel), но её старт ещё в будущем
  Когда клиент открывает экран
  Тогда эта запись показана в группе «Прошедшие», а не в «Предстоящие»
  И прошедшие заезды отсортированы по убыванию даты старта

Сценарий: AC-002 Отображение статуса записи заезда
  Дано запись имеет статус active, cancelled или late_cancel
  Когда клиент видит её карточку
  Тогда на карточке показан бейдж со статусом «активен», «отменен» или «поздняя отмена»
  И статус передан текстом и формой бейджа, а не только цветом

Сценарий: AC-003 Пустой список заездов
  Дано у клиента нет бронирований в системе
  Когда он открывает экран
  Тогда показана заглушка «У вас пока нет записей»
  И доступно действие «Записаться на заезд», ведущее к списку слотов (SCR-002)

Сценарий: AC-004 Только свои записи заездов
  Дано в системе есть бронирования разных пилотов
  Когда клиент открывает «Мои бронирования»
  Тогда он видит исключительно собственные записи заездов
  И на экране нет чужих или административных данных маршалов

Сценарий: AC-005 Переход к деталям брони заезда
  Дано на экране показана карточка записи заезда
  Когда клиент тапает по карточке
  Тогда открывается экран деталей брони (SCR-006)

Сценарий: AC-006 Обновление списка через pull-to-refresh
  Дано экран находится в состоянии Content
  Когда клиент выполняет жест pull-to-refresh (тянет список вниз)
  Тогда список заездов перезапрашивается через API и отображает актуальные данные и статусы
```

## 11. Решения

1. **Отменённые заезды — всегда в «Прошедшие»** (в том числе отмененные центром по погоде с будущим временем старта); секция «Предстоящие» содержит только активные заезды.[cite: 4]
2. **Сортировка прошедших слотов** — по убыванию даты старта (самые свежие заезды отображаются сверху).[cite: 4]
3. **Empty по секциям** — отдельная подсказка «Нет предстоящих записей» в секции «Предстоящие», если у пилота остались только прошедшие заезды; общий empty «У вас пока нет записей» выводится, когда нет ни одной брони.[cite: 4]
4. **При отсутствии истории прошедших** — секция/вкладка «Прошедшие» показывается пустой (не скрывается на UI), чтобы навигация по секциям оставалась предсказуемой для пользователя.[cite: 4]
5. **Глубина истории прошедших заездов** — параметр данных/конфига бэкенда; в MVP мобильное приложение запрашивает и грузит доступную историю целиком, лимит на клиенте не хардкодим.[cite: 4]
